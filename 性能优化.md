App 实例生命周期中存在长任务

App 实例生命周期函数中的长任务，如 onLaunch、onShow 等会同步阻塞首屏渲染，影响启动耗时。可通过延迟加载非关键代码的方式进行优化。

页面首屏渲染前多次调用同步 API，阻塞当前线程，影响启动耗时

小程序启动流程中，会注入开发者代码并顺序同步执行 App.onLaunch, App.onShow, Page.onLoad, Page.onShow。在小程序初始化代码（Page，App 定义之外的内容）和上述启动相关的几个生命周期中，应尽量减少或不调用同步 API。同步 API 虽然使用起来更简单，但是会阻塞当前 JS 线程，影响代码执行。如非必要，应尽可能的使用异步 API 代替同步，并将启动过程中非必要的同步 API 调用延迟到启动完成后进行。

页面生命周期函数中存在长任务

页面生命周期函数，例如 Page.onLoad 和 Page.onShow 会在首屏渲染前执行，影响首屏时间。页面切换时，会先调用前一个页面的 onHide 或 onUnload 生命周期，再进行新页面的创建和渲染。如果 onHide 和 onUnload 执行过久，可能导致页面切换的延迟，应避免在页面或组件生命周期函数中出现耗时任务。

重复获取相同 key 的 storage 信息

开发者在启动过程中可能会高频访问 storage 中的缓存信息。尤其是通过 getStorageSync 和 setStorageSync 接口，同步调用会阻塞 JS 线程，影响启动和页面渲染性能。对于未变更的内容，重复的 key 访问应使用内存缓存，避免无效调用。

访问页面过程中，存在声明但未被使用到的组件，如果将在特定条件下使用到，建议使用「用时注入」特性进行优化

启用按需注入后，页面 JSON 配置中定义的所有组件和 app.json 中 usingComponents 配置的全局自定义组件，都会被视为页面的依赖并进行注入和加载。建议开发者及时移除 JSON 中未使用自定义组件的声明，并尽量避免在全局声明使用率低的自定义组件，否则可能会影响按需注入的效果。在打开「按需注入」特性的前提下，可以通过「用时注入」特性使一部分自定义组件不在启动时注入，而是在真正被渲染时才进行注入，进一步降低小程序的启动和首屏时间。
